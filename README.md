
### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.
#### Ответ
Для данной цели подойдёт реляционная SQL база данных. Принцип ACID гарантирует целостноть и точность данных. 

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 
#### Ответ
 Есть несколько библиотек для хэширования OpenSSL, Intel IPP, xxHash 
1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.
#### Ответ
Для лендигов использовать или NoSQL СУБД Cassandra, MongoDB либо связку Redis Для кэша и Postgres c JSONB. Так как необходимо обеспечить быструю запись. и Гибкость данных. Структура данных так же может менятся в разных лендингах. 
Для CRM SQL по типу PostgreSQL проще делать сложные отчеты и контрроль целостности. Нет необходимости обеспечивать большую скорость на запись.


1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?
#### Ответ 
Можно связкой PostgreSQL + JSONB + Redis Редис будет как кэш при всплесках активности Использование JSONB ПОзволит гибко подходить к набору данных. А Сам Postgres это SQL база позволит делать сложную аналитику. 
1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.
#### Ответ 
Подойдёт иерархическая или документоориентированная СУБД Либо SQL с простой структурой ключевых полей 
1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?
Тот же PostgreSQL подойдет. Структура таблиц будет примерно следующего вида
IDОтдела NameОтдел
IDСотрудника ФИО IDОтдела
IDДокумента DATA IDСотрудника
где поле IDОтдела будет ключевым в списке отделов по нему мы будем строить соответствие в каком отделе работает сотрудник, а IDСотрудника Ключ по нему мы видим кто создал документ. И через рекурсивный запрос можнт построить всю структуру.


1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.
#### Ответ
Использовать специализированную графовую СУБД. Быстрая работа со связями (графовые алгоритмы: кратчайший путь, кластеризация точек, назначение курьеров).
Примеры СУБД Neo4j, ArangoDB.
1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?
#### Ответ
Можно подключить к той же графовой СУБД, если связи логистов и закупок пересекаются (например, поставщики → склады → маршруты).
Но лучше — сделать отдельную схему / граф для закупок и связать через интеграционную шину (API, Kafka).
Почему не всегда одна база:
У закупок другие паттерны нагрузки (больше транзакций по номенклатуре, меньше расчётов путей).
Безопасность: закупки не должны видеть курьерские маршруты и наоборот.
Разные SLA: логистике нужны доли секунды на расчёт пути, закупкам — консистентность заказов.
Если всё же одна СУБД — то ArangoDB (позволяет изолировать графы, разные коллекции, разный доступ).
1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?
#### Ответ
Да, можно.
Какую именно: PostgreSQL с расширениями.
Аргументация:
1.1 — нативная реляционная модель, ACID, сложные отчёты.
1.2 — JSONB для лендингов + полнотекстовый поиск для CRM.
1.3 — рекурсивные запросы для иерархии отделов.
1.4 — расширение pgRouting для графовых маршрутов, плюс можно моделировать связи.
Всё в одной базе → нет репликации между разными типами, единая безопасность, бэкапы.
Минус: сложность настройки и потенциальные проблемы с производительностью при очень высокой нагрузке (например, миллионы графовых вычислений в секунду). Но для крупной строительной компании — решаемо кластеризацией (Patroni, Citus).
Альтернатива: ArangoDB — мультимодельная (документы + графы + ключ-значение). Закрывает всё, кроме, возможно, сложной фин. отчётности с жёсткой целостностью (но можно добавить проверки на уровне приложения).
### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

#### Ответ

1. Начало транзакции Фиксируется время операции, идентификатор пользователя, сумма.
2. Проверка достаточности и блокировка средств на источнике
3. Отпарвка запроса оператору с данными клиента идентификатором транзакции и суммой.
4. Проверка оператором наличия клиента пополнение баланса.
5. При получении положительного ответа от оператора списание денежных средств формирование чека.
6. Снятие блокировки обновление оставшихся средств уведомление об успешном окончании платежа.

##### Правка задания транзакции на стороне оператора
1. Получение запроса от банка на зачисление по API http запросу или иному каналу связи
2. Начало транзакции в СУБД оператора запись запросf в буфер
3. Проверка валидности присланных данных существование номера итд
4. Увеличение баланса абонента в с случае корректности данных
5. Запись в лог платежей
6. Окончание транзакции отправка положительного кода ответа.


2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?
#### Ответ
Создается тригер который вызовет операцию обращение в банк на выполнение операции.
Банк проверит валидность данного запроса выпполнит операцию у себя вернет положительный код ответа
дальше аналогично 
В конце уведомление пользователя.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 
#### Ответ
1. Стандартизированный язык запросов. Один язык независимо от СУБД. 
2. Поддержка ACID 
3. Сложные JOIN и аналитические запросы
4. Строгая схема данных и целостность на уровне БД
5. Зрелость, инструментарий и сообщество
3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.
#### Ответ
Появляется горизонтальное масштабирование из "коробки" Классические SQL легко наращивают производительность только вертикально прямым увеличением мощности сервера. А вот Распределенные вычисления нет. Отсюда вытекает линейный рост при добавлении узлов. Из коробки адаптрованы под работу в облачных средах.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?
#### Ответ
Оценка типов данных 
Потребуется распределенная NewSQL СУБД.
Потребуется оценка соотношения оперцаий чтения/записи. Постоянно ли приходят новые данные или работа с архивом. И какое время ответа нас устроит. 
Универсальным будет Apache Spark 
Для большинства задач → Apache Spark (универсальный солдат: и пакетную обработку умеет, и потоковую, и SQL, и машинное обучение)
Если нужен ответ за доли секунды → ClickHouse (супербыстрая аналитическая база).Если нужно обрабатывать события в реальном времени (миллисекунды) → Apache Flink.
